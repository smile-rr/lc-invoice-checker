  Q3 — The missing value problem and what should actually happen

  This is the real design gap. Currently the Q1/Q2 gates in CheckExecutor use "any field present → proceed" logic. The problem has three dimensions:

  Problem A — "present but null" slips through
  hasAnyInvoiceField returns true if any invoice field is non-null. For a rule like UCP-18b-currency with invoice_fields: ["currency"] — if inv.currency = null, Q2 correctly fires → DISCREPANT via
  missing_invoice_action. But the description becomes the generic Q2 message ("Invoice is missing fields..."), not a meaningful discrepancy description.

  Problem B — missing ≠ mismatch, same DISCREPANT bucket
  Currently:
  - inv.currency = null → DISCREPANT with "Invoice is missing fields"
  - inv.currency = "EUR" when LC says "USD" → DISCREPANT with "Rule failed: {...}"

  Both are DISCREPANT but they describe completely different problems. A bank reviewer needs to know which one it is.

  Problem C — LC field "present in rawFields but empty as value"
  The Q1 gate checks lc.rawFields().containsKey(tag) — this is weaker than it looks. A tag can exist in rawFields but lc.currency can still be null if parsing produced an empty value. The SpEL expression then hits
  lc.currency.equalsIgnoreCase(...) → NullPointerException → caught as UNABLE_TO_VERIFY. Correct outcome, wrong route.
